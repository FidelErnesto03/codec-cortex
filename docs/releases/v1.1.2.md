## Problemas cerrados (v1.1.2)

### Fix 1 — Demo portable con exit 1 real

**Problema:** El demo v1.1.1 usaba `/home/z/my-project/.venv/bin/cortex` (ruta
absoluta del entorno del agente), fallaba en la verificación del usuario, pero
aun así imprimía "ALL RE-AUDIT GAPS FIXED".

**Solución:**
- `scripts/cortex_demo_v1_1_2.sh` localiza `cortex` dinámicamente:
  1. Si `cortex` está en `PATH`, lo usa.
  2. Si no, busca `src/cortex/` relativo al script y usa `python -m cortex`
     con `PYTHONPATH` seteado.
  3. Si no encuentra nada, falla con `exit 2` y mensaje claro.
- Helpers `expect_rc_zero` / `expect_rc_nonzero` cuentan fallos.
- El demo termina con `exit 1` real si `FAILURES > 0`.

### Fix 2 — recover --embed-aud-rsk declara AUD/RSK en $0

**Problema:** `recover --embed-aud-rsk` insertaba entradas `AUD:recovery` pero
si el `$0` fue reconstruido, `AUD` no quedaba declarado en el glosario. Luego
`verify --strict` fallaba con `E003_UNKNOWN_SIGIL`.

**Solución:**
- `_embed_recovery_trace()` ahora asegura que `AUD` y `RSK` estén declarados
  en `$0` antes de insertar las entradas.
- Si faltan, se auto-declaran con metadata canónica de `brain_sigils()` y se
  emite un diagnóstico `I003_SIGIL_AUTO_DECLARED`.
- El flujo `recover --embed-aud-rsk` + `verify --strict` pasa limpiamente.

### Fix 3 — Test recover + embed + verify --strict

**Tests añadidos:**
- `test_recover_embed_then_verify_strict` (API interna)
- `test_recover_embed_then_verify_strict_cli` (CLI end-to-end)

Ambos verifican que el artefacto recuperado con AUD/RSK embebidos pasa
`verify --strict` sin errores.

### Fix 4 — Validación post-mutación para update/delete/move

**Problema:** Solo `add` tenía el gate post-mutación. `update` podía dejar
inválido un `brain.cortex` sin bloquear la escritura.

**Solución:**
- `cortex.cli.commands.post_mutation_gate()` helper compartido que valida
  el AST después de una mutación y bloquea la escritura si hay errores
  (a menos que `--force`).
- `update`, `delete`, `move` ahora usan el mismo gate que `add`.
- Nuevos flags CLI para los tres comandos: `--no-validate-write`,
  `--strict-write`.

**Tests añadidos:**
- `test_cli_update_blocked_by_validation`
- `test_cli_update_force_bypasses_gate`
- `test_cli_delete_blocked_by_validation`
- `test_cli_move_post_mutation_validation`
- `test_internal_post_mutation_gate_helper`

### Fix 5 — diff --profile governance detecta cambios reales

**Problema:** `diff --profile governance` comparaba solo `findings` de
`validate_level_policy`. Dos archivos válidos con un `CNST` cambiado → rc=0
(falso negativo).

**Solución:**
- `_extract_governance_changes()` filtra los diffs de `compare_ast()` para
  quedarse solo con los que afectan sigilos de gobernanza (`FCS`, `OBJ`,
  `WRK`, `STP`, `NXT`, `CNST`, `AXM`, `!`, `RSK`, `AUD`, `CLAIM`, `LIM`,
  `KNW`).
- Retorna non-zero si hay cualquier cambio en entradas de gobernanza,
  además de si hay findings.

**Tests añadidos:**
- `test_diff_governance_detects_cnst_change`
- `test_diff_governance_equal_files_returns_zero`
- `test_diff_governance_ignores_non_governance_changes`

### Fix 6 — BENCHMARK.md sin rutas absolutas ni conteo hardcoded

**Problema:** BENCHMARK.md usaba rutas absolutas (`/home/z/...`) y mantenía
una referencia a `88/88 tests` aunque la suite actual es 110+ (luego 124).

**Solución:**
- Eliminadas todas las rutas `/home/z/...`.
- Eliminada la referencia hardcodeada a `88/88`.
- El archivo declara explícitamente que el conteo no se hardcodea y apunta
  al comando `python -m pytest src/tests/ -q` para obtener el número actual.

**Tests añadidos:**
- `test_benchmark_no_absolute_paths`
- `test_benchmark_roundtrip_count_matches_suite` (redefinido)

### Fix 7 — --json produce JSON real en new y render

**Problema:** `--json new` y `--json render` no producían JSON real (salida
en texto plano).

**Solución:**
- `cortex --json new ...` ahora emite JSON parseable con `{ok, path, bytes,
  kind}`.
- `cortex --json render ...` ahora emite JSON con `{ok, input, mode,
  markdown}` cuando no hay `--out`, o `{ok, out, bytes}` cuando hay `--out`.
- README actualizado: aclara que `--json` es un flag global que va **antes**
  del subcomando (`cortex --json new ...`).

**Tests añadidos:**
- `test_cli_new_json_produces_valid_json`
- `test_cli_render_json_produces_valid_json`
- `test_cli_render_json_to_file`

## Instalación y verificación

```bash
# Extraer el tarball en cualquier directorio
tar xzf codec-cortex-1.1.2.tar.gz
cd cortex

# Instalar (editable)
pip install -e .

# Verificar tests (124/124)
python -m pytest src/tests/ -q

# Ejecutar demo portátil (debe terminar con exit 0)
bash scripts/cortex_demo_v1_1_2.sh
echo "Demo exit code: $?"  # debe ser 0

# Verificar madurez
cortex --version            # cortex 1.1.2
cortex new brain --out /tmp/brain.cortex --force
cortex verify /tmp/brain.cortex --strict --kind brain
```

## Juicio técnico

La v1.1.2 cierra los 7 problemas puntuales detectados por el usuario en la
verificación de v1.1.1, sin rediseñar la arquitectura. Los fixes son
quirúrgicos:

1. **Demo portátil** — localiza `cortex` dinámicamente, exit 1 real en fallo.
2. **recover --embed-aud-rsk** — AUD/RSK auto-declarados en $0 reconstruido.
3. **Tests recover+verify --strict** — cubren el flujo completo.
4. **Validación post-mutación** — `post_mutation_gate()` compartido por
   add/update/delete/move.
5. **diff governance** — detecta cambios reales en entradas de gobernanza,
   no solo findings.
6. **BENCHMARK.md** — sin rutas absolutas ni conteo hardcoded.
7. **--json** — produce JSON real en new y render.

La suite de tests creció de 110 → 124 (14 nuevos tests que verifican cada
fix). El demo portátil funciona tanto con `cortex` instalado en PATH como
sin instalación (vía PYTHONPATH), y termina con exit code non-zero si
cualquier verificación falla.
