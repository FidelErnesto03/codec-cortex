# HCORTEX-AUDIT — Verificación ejecutable codec-cortex v2.0.0

## $1 · Identidad de auditoría

| Campo | Valor |
|---|---|
| Artefacto auditado | `codec-cortex-2.0.0.tar.gz` |
| Informe auditado | `INFORME_DE_ENTREGA_v2.0.0.md` |
| Fecha de auditoría | 2026-06-29 |
| Modo | ejecución real + pruebas adversariales |
| Criterio | SKILL.md / CODEC-CORTEX: honestidad de madurez, `$0`, recuperación, roundtrip, no overclaim |

## $2 · Resultado ejecutivo

**Dictamen:** NO aprobar v2.0.0 como enterprise-candidate.

v2.0.0 introduce un módulo `cortex.v2` útil y el fixture `SKILL_v2.cortex.md` sí pasa roundtrip textual exacto. Sin embargo, la entrega no conserva la línea base v1.1.9, reduce cobertura de tests, reabre brechas ya cerradas, y el claim `byte-identical` es técnicamente incorrecto porque el comando compara texto normalizado, no bytes reales.

## $3 · Evidencia ejecutada

### Instalación y suite oficial

```text
cortex --version
→ cortex 2.0.0

python -m pytest src/tests/ -q
→ 207 passed in 6.04s
```

### Roundtrip v2 declarado

```text
cortex v2-roundtrip src/tests/fixtures/SKILL_v2.cortex.md
→ ✓ Roundtrip byte-identical
→ sections: 13
→ entries: 222
→ bytes: 37399
```

### Roundtrip contra `SKILL_canon.md` incluido

```text
cortex v2-roundtrip SKILL_canon.md
→ ✗ Roundtrip FAILED
→ original: 1625 lines
→ reproduced: 40 lines
→ first diff line 1
```

Este resultado no invalida el fixture v2, pero sí muestra que el claim “con SKILL.md” es ambiguo. El comando funciona sobre `src/tests/fixtures/SKILL_v2.cortex.md`, no sobre el `SKILL_canon.md` incluido en el paquete.

## $4 · Hallazgos críticos

### H-01 · Regresión desde v1.1.9

El tarball v2.0.0 solo trae tests hasta `test_v1_1_7_fixes.py` y `test_v2_roundtrip.py`. No trae los tests de v1.1.8 ni v1.1.9.

Al copiar y ejecutar los tests de v1.1.8/v1.1.9 contra v2.0.0:

```text
python -m pytest test_v1_1_8_fixes.py test_v1_1_9_fixes.py -q
→ 24 failed, 2 passed
```

Fallaron específicamente:

- recovery con `$1` y `$99` existentes;
- recovery sin contaminación de sección existente;
- RSK embebido para estado vivo movido desde `$0`;
- AUD de recovery describiendo evento real;
- null-like sentinels: `none`, `nil`, `undefined`, `n/a`, `tbd`, `???`, `-`, `--`;
- recovery de `$0` existente pero incompleto;
- recovery CLI que debería reparar glosario incompleto y retornar `0`.

**Impacto:** v2.0.0 no es una evolución segura de v1.1.9. Es una rama v2 basada aproximadamente en v1.1.7, con regresión de hardening.

### H-02 · Claim `byte-identical` técnicamente falso

El comando `v2-roundtrip` abre archivos en modo texto (`open(..., "r", encoding="utf-8")`) y compara strings. Eso normaliza saltos de línea y no compara bytes crudos.

Prueba con una copia CRLF del fixture:

```text
archivo contiene CRLF reales: True
cortex v2-roundtrip SKILL_v2_crlf.cortex.md --format json
→ ok: true
→ roundtrip: byte-identical
```

Pero los bytes reales no son iguales a los que produciría el writer con `\n`.

Además, el comando reporta:

```text
bytes: 37399
```

pero el archivo real tiene:

```text
stat bytes: 38018
len(bytes): 38018
len(chars): 37399
```

**Conclusión:** el roundtrip puede llamarse `text-identical` o `normalized-text-identical`, no `byte-identical`.

### H-03 · Recovery incompleto sigue fallando

Caso probado:

```text
$0: GLOSSARY
# IDN | identity | attrs | B | Semantic | Identity

$1: CONTENT
IDN:package{name:"legacy"}
KNW:topic{topic:"x", content:"y", status:"current"}
```

Resultado:

```json
{
  "ok": false,
  "reconstructed_glossary": false,
  "diagnostics": [],
  "post_validation_errors": [
    {"code": "E003_UNKNOWN_SIGIL", "sigil": "KNW"},
    {"code": "E003_UNKNOWN_SIGIL", "sigil": "KNW"}
  ],
  "conformant": false
}
```

Este era precisamente el cierre esperado en v1.1.9: reparar `$0` existente pero incompleto y declarar sigilos canónicos usados pero ausentes.

### H-04 · Null-like sentinels vuelven a pasar como válidos

Prueba directa con archivo generado por `cortex new brain`:

```bash
cortex update brain.cortex CNST:self_contained --set rule=none --force
cortex verify brain.cortex --strict --kind brain
```

Resultado:

```text
update rc=0
verify errors=0
CNST:self_contained{rule:"none", severity:"blocking", survive:"min"}
```

Esto reabre la brecha semántica: una restricción crítica puede existir formalmente, pero carecer de contenido real.

### H-05 · Parser/writer v2 no preserva información no modelada

El parser v2 omite líneas no reconocidas. Si se agrega un comentario entre entradas:

```text
# comment line
```

`v2-roundtrip` falla porque el writer reconstruye desde AST y elimina el comentario.

Si se agrega un header `notes: "hello world"`, el parser conserva las comillas y el writer vuelve a quote-ea el valor:

```text
orig:  notes: "hello world"
repro: notes: ""hello world""
```

**Impacto:** el parser/writer v2 no es un codec general de preservación textual. Es un serializador ajustado al fixture conocido.

## $5 · Hallazgos medios

### M-01 · Conteo de tests engañoso respecto de continuidad

El informe declara:

```text
207/207 pasan (196 heredados + 11 nuevos v2)
```

Pero v1.1.9 tenía 222 tests en la entrega anterior. v2.0.0 no hereda los tests de v1.1.8/v1.1.9. Por tanto, `207/207` no representa continuidad total; representa una base anterior + v2.

### M-02 · `SKILL.md` externo y `SKILL_canon.md` siguen generando ambigüedad

La conversación contiene un `SKILL.md` v1.1.0-enterprise-candidate, mientras v2.0.0 declara `SKILL.md v1.2.0-enterprise-candidate`. El tarball incluye `SKILL_canon.md`, pero el roundtrip v2 no aplica a ese archivo, sino al fixture `SKILL_v2.cortex.md`.

### M-03 · HCORTEX v2 está correctamente marcado como pendiente, pero v2.0.0 no debe sugerir consolidación enterprise

El informe reconoce que CORTEX → HCORTEX y viceversa requiere v2.1.0. Esto es honesto, pero refuerza que v2.0.0 no debe declararse como enterprise-stable.

## $7 · Recomendación de decisión

No aceptar v2.0.0 como enterprise-candidate.

Aceptar solo como rama experimental de parser/writer CORTEX v2 con valor técnico puntual.

### Condición para v2.0.1

1. Rebasear v2.0.0 sobre v1.1.9 real.
2. Restaurar `test_v1_1_8_fixes.py` y `test_v1_1_9_fixes.py`.
3. Lograr suite completa: `222 + tests v2`, no `196 + tests v2`.
4. Corregir `v2-roundtrip` para comparar bytes reales (`Path.read_bytes()`), no strings normalizados.
5. Reportar `bytes` reales, no `len(str)`.
6. Cambiar etiqueta a `text-identical` si se decide mantener comparación textual.
7. Aclarar que `SKILL_v2.cortex.md` es fixture CORTEX v2, no `SKILL_canon.md` ni `SKILL.md` humano.
8. Agregar pruebas v2 adversariales: CRLF, header notes quoted, comentarios, líneas no reconocidas, unicode byte-count.
9. Mantener HCORTEX v2 como `planned` hasta implementación real.
