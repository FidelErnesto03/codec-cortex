# RSK · Brechas encontradas

## RSK:H-01 · `recover --embed-aud-rsk` no cubre todos los archivos sin `$0`

### Resultado

El informe afirma que `recover --embed-aud-rsk` declara `AUD/RSK` en `$0` y que el flujo `recover + verify --strict` pasa limpiamente. Esto es cierto cuando el archivo legacy empieza por una sección distinta de `$0`, por ejemplo `$1`. Pero falla cuando el archivo sin `$0` empieza directamente con entradas `.cortex`.

### Prueba adversarial

Entrada legacy:

```text
IDN:package{name:"legacy", kind:"package", status:"current"}
KNW:topic{topic:"x", content:"y", status:"current"}
```

Comandos:

```bash
cortex recover legacy_no_gloss.cortex --out recovered_embed.cortex --embed-aud-rsk
cortex verify recovered_embed.cortex --strict
```

Resultado observado:

```text
recover_rc=0
verify_rc=1
errors: 4
[E003_UNKNOWN_SIGIL] sigil 'IDN' not declared in $0
[E003_UNKNOWN_SIGIL] sigil 'KNW' not declared in $0
```

### Diagnóstico

El parser acepta el archivo como una estructura con `$0: GLOSSARY` vacío o equivalente, no dispara correctamente la reconstrucción por ausencia real de `$0`, y `recover` no reconstruye sigilos observados. Esto rompe recuperación segura.

### Impacto

Alto. El SKILL exige que un archivo sin `$0` sea tratado como no conforme, leído solo en recuperación, reconstruido con `$0` mínimo, marcado con `RSK/AUD` y reemitido antes de ser usado como memoria confiable.

### Recomendación

Modificar `recover_cortex()` para detectar glosario vacío con entradas observables como equivalente a missing-glossary. Condición sugerida:

```text
if glossary.sigils == empty and observed_entry_starts > 0:
    reconstruct_glossary()
```

Agregar test CLI:

```text
test_recover_entry_first_file_without_glossary_then_verify_strict
```

## RSK:H-02 · CRUD persiste entradas críticas incompletas por defecto

### Resultado

`add` tiene post-mutation validation, pero por defecto no bloquea warnings. Como los campos requeridos de sigilos críticos se reportan como `W001_MISSING_FIELDS`, el CLI permite persistir entradas no conformes salvo que el usuario use `--strict-write`.

### Prueba adversarial

Comando:

```bash
cortex add brain.cortex --section '$2' --sigil OBJ --name partial --value 'goal:"partial"'
cortex verify brain.cortex --strict --kind brain
```

Resultado observado:

```text
add_rc=0
written.ok=true
diagnostics: W001_MISSING_FIELDS
verify_strict_rc=1
[W001_MISSING_FIELDS] OBJ:partial missing required fields: ['status', 'success', 'survive']
```

### Diagnóstico

El contrato del SKILL llama “campos requeridos” a esos campos. En una implementación gobernada, una entrada `OBJ` sin `status`, `success` y `survive` no debería persistirse por defecto.

### Impacto

Alto. Permite que el CLI cree memoria aparentemente válida en modo normal, pero inválida en modo estricto. Esto debilita el claim de “CRUD gobernado”.

### Recomendación

Convertir `W001_MISSING_FIELDS` en error para sigilos críticos (`FCS`, `OBJ`, `WRK`, `STP`, `CNST`, `RSK`, `AUD`, `CLAIM`, `LIM`, `SES`, `LNG`, `KNW`) o hacer que el post-mutation gate use `strict=True` por defecto.

## RSK:H-03 · `--force` permite persistir secretos en claro

### Resultado

Sin `--force`, el secret scanning bloquea correctamente. Con `--force`, el CLI persiste el secreto aunque `verify --strict` luego falle.

### Prueba adversarial

Comando:

```bash
cortex add secret_force.cortex \
  --section '$3' \
  --sigil REF \
  --name secret \
  --value 'provider:"x", password:"abc123456"' \
  --force
```

Resultado observado:

```text
add_secret_force_rc=0
verify_rc=1
[E028_SECRET_IN_CLEAR] possible password in clear text in REF:secret
```

### Diagnóstico

`--force` es útil para recovery estructural, pero no debería poder anular una prohibición de seguridad. El SKILL dice que `.cortex` MUST NOT almacenar secretos en claro.

### Impacto

Alto. Riesgo de seguridad real y contradicción directa con el contrato de cumplimiento.

### Recomendación

Marcar `E028_SECRET_IN_CLEAR` como no bypassable. Si se requiere recovery forense, exigir un modo explícito separado, por ejemplo:

```text
--unsafe-allow-secret-forensics
```

Y que nunca sea el comportamiento ordinario de `--force`.

## RSK:H-04 · `FCS/OBJ status:done` cuenta como activo

### Resultado

Un `brain.cortex` con `FCS` y `OBJ` en `status:"done"` pasa `verify --strict --kind brain`.

### Prueba adversarial

Comando:

```bash
cortex update brain_done.cortex FCS:primary --set status=done
cortex update brain_done.cortex OBJ:main --set status=done
cortex verify brain_done.cortex --strict --kind brain
```

Resultado observado:

```text
rc=0
errors: 0
warnings: 0
```

### Diagnóstico

La implementación considera `{current, blocked, done}` como estados activos para validar `FCS` y `OBJ`. Esto es semánticamente débil: `done` indica cierre, no foco activo ni objetivo operativo vigente.

### Impacto

Medio/alto. Un cerebro operativo podría quedar “válido” pero sin objetivo activo real.

### Recomendación

Cambiar `active_status` a:

```text
current | blocked
```

Y tratar `done` como histórico/cerrado, no como activo.

## RSK:M-01 · HCORTEX no declara `Perfil:` como primera línea real

### Resultado

El SKILL exige que la primera línea del render HCORTEX sea `Perfil: CORTEX-<LEVEL>`. La salida actual empieza con un comentario HTML:

```text
<!-- cortex-render: hcortex-read; roundtrip: false -->

Perfil: CORTEX-RECOVERY
```

### Impacto

Medio. No rompe legibilidad, pero sí rompe una regla textual explícita útil para parsers, auditoría y chequeo humano rápido.

### Recomendación

Mover el comentario debajo de la línea de perfil:

```text
Perfil: CORTEX-RECOVERY
<!-- cortex-render: hcortex-read; roundtrip: false -->
```

## RSK:M-02 · `render --mode audit` se autodescribe como `Mode: READABLE`

### Resultado

Ejecutar:

```bash
cortex render brain.cortex --mode audit
```

produce un encabezado interno:

```text
Mode: READABLE
```

aunque sí activa `with_source`.

### Impacto

Medio. La trazabilidad puede estar presente, pero el metadato humano/auditable declara el modo incorrecto.

### Recomendación

Pasar `mode="AUDIT"` a `render_hcortex_read()` cuando el usuario usa `--mode audit`.

## RSK:M-03 · `source` en AUDIT no es uniforme para `cuerpo`/`bloque`

### Resultado

Para entradas `attrs`, audit agrega columna `source`. Para `AXM` (`cuerpo`) y `DIAG` (`bloque`), el source aparece solo como comentario HTML de sección:

```text
<!-- section: $2 · AXM:principle · P0 -->
```

pero no como columna o marca visible en el bloque.

### Impacto

Medio. Puede ser suficiente para auditoría técnica cruda, pero no cumple literalmente la regla “agregar columna `source` con `<SIGIL>:<name>`” ni es homogéneo para todos los tipos.

### Recomendación

Para `cuerpo`/`bloque`, agregar una línea visible antes del bloque:

```markdown
source: `AXM:principle`
```

## RSK:M-04 · `decode` alias no sigue la UX canónica planificada

### Resultado

El SKILL planifica:

```text
cortex decode <file> [--format yaml|hcortex]
```

Pero en v1.1.2:

```bash
cortex decode brain.cortex
```

falla porque exige `--mode`.

### Impacto

Medio/bajo. Es una brecha de contrato de interfaz, no del núcleo. Pero si se declara “alias CLI canónico”, debería tener un default razonable, por ejemplo `--mode readable`.

### Recomendación

Permitir:

```bash
cortex decode brain.cortex
```

como alias de:

```bash
cortex render brain.cortex --mode readable
```

## RSK:M-05 · Claim documental excesivo sobre recuperación legacy

### Resultado

`STATUS.md` declara recuperación legacy como `current (1.1.2)` y el informe afirma que `recover --embed-aud-rsk + verify --strict` pasa limpiamente. La prueba adversarial demuestra que esto no es universal: falla si el archivo sin `$0` empieza directamente con entradas.

### Impacto

Medio. La capacidad existe, pero debería declararse como current con límites explícitos, no como cierre total.

### Recomendación

Cambiar el claim a:

```text
current with known limits
```

o cerrar H-01 antes de mantenerlo como `current` sin salvedad.

### Resultado

La v1.1.2 es una mejora real y corrige buena parte de los defectos señalados en v1.1.1. Aun así, no debe declararse “sin brechas” ni plenamente conforme.

Dictamen:

```text
ACEPTACIÓN CONDICIONADA.
La base es sólida, pero requiere v1.1.3 antes de declarar conformidad fuerte CODEC-CORTEX.
```

El problema ya no es arquitectura general. El problema está en los bordes de gobernanza: recuperación incompleta, bypass de seguridad, persistencia de contratos críticos incompletos y metadatos HCORTEX que no cumplen literalmente el SKILL.
