## $3 · VEREDICTO

ACEPTACIÓN CONDICIONADA.

La v1.1.3 cierra las brechas P0/P1/P2 declaradas en el informe y mejora sustancialmente la correspondencia con SKILL. Sin embargo, encontré brechas nuevas de gobernanza y empaquetado que impiden declarar cierre pleno.

La brecha principal no está en `verify`: está en el **gate de escritura con `--force`**. El validador detecta un `brain.cortex` sin FCS/OBJ activo, pero el CRUD todavía puede persistir ese estado inválido si el usuario fuerza la mutación. Esto contradice la idea de que el Nivel 2 debe conservar foco y objetivo como condición operativa.

## $5 · BRECHAS ENCONTRADAS

### H-113-01 · `--force` puede persistir un brain sin FCS activo

Severidad: ALTA.

Prueba:

```bash
cp brain.cortex brain_delete_fcs.cortex
cortex delete brain_delete_fcs.cortex FCS:primary --force
cortex verify brain_delete_fcs.cortex --strict --kind brain
```

Resultado:

```text
DEL_FCS_FORCE_RC=0
ok: true
E024_LEVEL2_MISSING_FOCUS aparece en diagnostics, pero el archivo fue escrito
verify posterior: rc=1
```

Impacto:

- El CLI detecta el error, pero permite persistir memoria operativa inválida.
- `E024_LEVEL2_MISSING_FOCUS` sigue siendo bypassable con `--force`.
- Esto abre una brecha de gobierno: el usuario puede dejar un `brain.cortex` sin foco activo.

Criterio SKILL afectado:

- Nivel 2 debe contener foco y objetivo.
- El agente debe validar FCS/OBJ antes de actuar.
- Los pitfall empresariales exigen no truncar/eliminar P0.

Recomendación:

`E024_LEVEL2_MISSING_FOCUS` debe ser `bypassable=False` en mutaciones de `kind=brain`, al menos para `delete`, `update`, `move` y cualquier operación que deje FCS/OBJ sin estado activo. Si se necesita fabricar fixtures inválidos, debe hacerse con un modo explícito `--unsafe-write-invalid-fixture`, no con `--force` operativo.

### H-113-02 · `--unsafe-allow-secret-forensics` persiste secretos y deja el archivo inválido

Severidad: ALTA/MEDIA, según alcance declarado.

Prueba:

```bash
cortex add brain2.cortex --section '$3' --sigil REF --name secret_unsafe \
  --value 'password:"abc123456"' --create-section --unsafe-allow-secret-forensics
cortex verify brain2.cortex --strict --kind brain
```

Resultado:

```text
SECRET_UNSAFE_RC=0
verify posterior: rc=1
E031_SECRET_NOT_BYPASSABLE
```

Impacto:

- El nombre del flag es honesto (`unsafe`), pero el comando responde `ok:true` y escribe un `.cortex` que viola la regla de secretos.
- En términos de SKILL, ese artefacto no debe considerarse conforme ni operativo.

Recomendación:

Mantener el bypass solo si:

1. escribe a un artefacto de cuarentena, no al `brain.cortex` normal;
2. agrega `LIM`/`RSK` visible indicando `non_conformant_forensic_artifact`;
3. obliga `--out` distinto del archivo fuente;
4. `verify --strict` debe seguir fallando, como ya ocurre.

### M-113-03 · La instalación documentada no reproduce tests sin instalar pytest

Severidad: MEDIA.

El informe dice:

```bash
pip install -e .
python -m pytest src/tests/ -q
```

En venv limpia, después de `pip install -e .`, el comando falla:

```text
No module named pytest
```

Luego de `pip install pytest`, la suite pasa. Esto no afecta el CLI, pero sí la reproducibilidad declarada.

Recomendación:

Agregar dependencia de desarrollo:

```toml
[project.optional-dependencies]
dev = ["pytest"]
```

Y documentar:

```bash
pip install -e '.[dev]'
python -m pytest src/tests/ -q
```

### M-113-05 · Recovery entry-first no inserta RSK si los sigilos reconstruidos son canónicos

Severidad: MEDIA/BAJA.

En un archivo entry-first con `IDN` y `KNW`, el recovery declara `RSK` en `$0`, pero solo inserta `AUD:recovery`; no inserta `RSK:*`. Para un sigilo desconocido (`FOO`), sí inserta `RSK:reconstructed_foo`.

Esto es defendible si se interpreta que los sigilos canónicos no son ambiguos. Pero el archivo sin `$0` completo sí sigue siendo un caso de recuperación con riesgo semántico.

Recomendación:

Agregar un `RSK:recovered_without_glossary` general cuando `E030_RECOVERY_INCOMPLETE` esté presente, aunque los sigilos sean canónicos.
