## 3. Brecha crítica encontrada

### H-114-01 — Entradas operativas dentro de `$0` son aceptadas, usadas parcialmente y luego ocultadas por HCORTEX

#### Severidad

`Alta / P0 conceptual`.

#### Norma afectada

El SKILL define que todo `.cortex` debe iniciar con `$0`, que `$0` es la fuente estructural local para sigilos, tipos, contratos y reglas, y aclara expresamente que el contenido de `$0` no se interpreta como memoria de trabajo. Por tanto, una entrada como `FCS:primary{...}` dentro de `$0` no debería contar como foco operativo.

#### Prueba adversarial

Archivo construido:

```text
$0: BAD GLOSSARY
# Sigil | Name | Type | Risk | Cognitive Layer | Description
# IDN | identity | attrs | B | Semantic | Identity
# FCS | focus | attrs | H | Working | Focus
# OBJ | objective | attrs | H | Working | Objective
IDN:agent{name:"bad-brain"}
FCS:primary{what:"x", priority:"high", status:"current", survive:"min"}
OBJ:main{goal:"y", status:"current", success:"z", survive:"min"}
```

Comando:

```bash
cortex verify brain_all_under_zero.cortex --strict --kind brain
```

Resultado:

```text
errors: 0
warnings: 0
entries: 0
```

El resultado es contradictorio: `verify` acepta el archivo como brain válido aunque reporta `entries: 0`. Al mismo tiempo, `cortex list` sí enumera `IDN`, `FCS` y `OBJ` dentro de `$0`.

#### Impacto

El CLI permite crear un brain que parece válido para `verify`, pero queda vacío para HCORTEX porque HCORTEX omite `$0` por defecto. Esto produce una memoria con foco y objetivo “fantasma”: existen para algunas rutas internas, pero desaparecen en la vista humana auditable.

#### Evidencia adicional

Comando:

```bash
cortex render brain_all_under_zero.cortex --profile full --mode audit
```

Resultado:

```text
Perfil: CORTEX-FULL
# HCORTEX-READ
> ... Mode: AUDIT ...
```

No aparecen `FCS`, `OBJ` ni `IDN`, aunque `verify --kind brain` los aceptó como suficientes.

#### Conclusión

Esto rompe la separación entre `$0` y memoria. El validador no debe permitir entradas de memoria operacional en `$0`, y mucho menos usarlas para satisfacer invariantes de `brain.cortex`.

## 4. Brecha crítica derivada en recovery

### H-114-02 — `recover` de archivo entry-first reubica contenido recuperado dentro de `$0`

#### Severidad

`Alta`.

#### Contexto

La v1.1.4 declara que `recover` reconstruye `$0` y agrega `RSK:reconstructed_glossary` incluso para sigilos canónicos. Eso se cumple parcialmente: reconstruye `$0`, declara `AUD/RSK` y pasa `verify --strict`.

Pero si el archivo original empieza directamente con entradas, el contenido queda en la sección `$0` del archivo reparado.

#### Prueba adversarial

Archivo legacy:

```text
IDN:package{name:"legacy"}
KNW:topic{topic:"x", content:"y", status:"current"}
```

Comando:

```bash
cortex recover legacy_entry_first.cortex --out recovered_entry_first.cortex --embed-aud-rsk
```

Archivo recuperado relevante:

```text
$0: RECOVERED GLOSSARY
...
IDN:package{name:"legacy"}
KNW:topic{topic:"x", content:"y", status:"current"}

$8: RECOVERY AUDIT
AUD:recovery{...}

$5: RECOVERY RISKS
RSK:reconstructed_glossary{...}
```

#### Resultado de verificación

```bash
cortex verify recovered_entry_first.cortex --strict
```

Resultado:

```text
errors: 0
warnings: 0
```

#### Resultado de render

```bash
cortex render recovered_entry_first.cortex --profile full --mode audit
```

Resultado observado:

```text
AUD:recovery visible
RSK:reconstructed_glossary visible
IDN:package no visible
KNW:topic no visible
```

#### Impacto

El recovery dice reparar el archivo, pero deja contenido semántico recuperado en `$0`, que HCORTEX omite por diseño. Por eso la recuperación preserva sintaxis, pero pierde visibilidad humana y auditabilidad del contenido original.

#### Conclusión

`recover` no debe insertar `$0` encima de entradas entry-first sin crear una sección operacional posterior. Debe reubicar el payload original en una sección explícita, por ejemplo:

```text
$1: RECOVERED CONTENT
IDN:package{...}
KNW:topic{...}
```

## 5. Brecha de CRUD

### H-114-03 — `cortex add --section $0` permite esconder estado operacional

#### Severidad

`Alta`.

#### Prueba adversarial

Comando:

```bash
cortex add add_to_zero.cortex --section '$0' --sigil NXT --name bad \
  --value 'action:"x", trigger:"y", status:"current", survive:"min"' --force
```

Resultado:

```json
{
  "ok": true,
  "section": "$0",
  "sigil": "NXT",
  "name": "bad"
}
```

Luego:

```bash
cortex verify add_to_zero.cortex --strict --kind brain
```

Resultado:

```text
errors: 0
warnings: 0
```

Y:

```bash
cortex render add_to_zero.cortex --mode audit --profile full
```

Resultado: `NXT:bad` no aparece.

#### Impacto

Esto permite persistir acciones en cola invisibles para HCORTEX, sin error de validación. Es exactamente el tipo de contaminación que CODEC-CORTEX busca evitar: memoria operativa no auditable.

#### Conclusión

Debe existir una regla non-bypassable:

```text
E033_ZERO_SECTION_MEMORY_ENTRY
$0 MUST NOT contain operational entries; only glossary declarations/metadata are allowed.
```

## 6. Brecha documental de benchmark

### M-114-04 — Claims `measured` no cubren el borde adversarial encontrado

#### Severidad

`Media`.

#### Observación

`BENCHMARK.md` declara `Recovery completeness` como `measured` y `Recovery + embed AUD/RSK + verify --strict` como `measured`, pero la prueba adversarial entry-first demuestra que `verify --strict` puede pasar aunque el contenido recuperado quede invisible en HCORTEX.

#### Impacto

El benchmark mide “pasa verify”, pero no mide “contenido recuperado queda fuera de `$0` y visible en HCORTEX”. La métrica actual no captura completitud semántica ni auditabilidad.

#### Recomendación

Reclasificar o endurecer la métrica:

```text
Recovery completeness = measured only si:
1. verify --strict pasa;
2. list muestra las entradas recuperadas fuera de $0;
3. render --mode audit --profile full muestra esas entradas;
4. RSK/AUD quedan visibles y declarados.
```

## 8. Recomendación para v1.1.5

### P0 — Obligatorio antes de aprobación plena

1. Prohibir entradas operativas dentro de `$0`.
2. `verify --strict` debe fallar si cualquier entrada no-glosario aparece en `$0`.
3. `brain` no debe poder satisfacer `FCS/OBJ` con entradas ubicadas en `$0`.
4. HCORTEX puede seguir omitiendo `$0`, pero debe advertir si detecta entradas no-glosario allí.
5. `recover` entry-first debe crear `$1: RECOVERED CONTENT` antes de reemitir entradas originales.
6. `cortex add --section $0` debe quedar limitado a operaciones de glosario explícitas, no a entradas de memoria.

### P1 — Endurecimiento de pruebas

Agregar tests adversariales:

```text
test_verify_rejects_memory_entries_in_zero_section
test_brain_fcs_obj_under_zero_do_not_satisfy_level2
test_recover_entry_first_moves_payload_to_recovered_content_section
test_hcortex_audit_shows_recovered_entry_first_payload
test_add_to_zero_rejects_operational_entry
test_benchmark_recovery_requires_hcortex_visibility
```

### P2 — Benchmark y documentación

Actualizar `BENCHMARK.md`:

```text
Recovery completeness debe medir conservación semántica y visibilidad HCORTEX, no solo verify --strict.
```

v1.1.4 es la mejor versión hasta ahora. Corrige problemas reales de gobernanza y seguridad. Pero la brecha `$0` es de arquitectura, no de cosmética: permite que memoria operativa exista en un lugar que el propio SKILL declara como metadato estructural y que HCORTEX omite por defecto.

Por eso mi recomendación es:

```text
No regresar.
No aprobar plenamente.
Exigir v1.1.5 enfocada exclusivamente en sección $0, recovery entry-first y benchmark de visibilidad HCORTEX.
```
