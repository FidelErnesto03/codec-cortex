## Resultado de la comparación

**No es posible declarar científicamente un ganador con este benchmark tal como está construido.** Sí permite comparar densidad y cualidades estructurales, pero la medición de precisión y eficiencia está contaminada por errores metodológicos y semánticos.

### 1. Densidad real

Medí únicamente el contenido de cada representación, excluyendo los encabezados externos y las cercas Markdown:

| Formato             | Caracteres | Tamaño frente a CORTEX | Líneas |
| ------------------- | ---------: | ---------------------: | -----: |
| **Markdown Tables** |  **4.099** |            **−12,5 %** |     76 |
| **CORTEX 0.1**      |      4.684 |             referencia | **56** |
| Plain Text          |      5.984 |                +27,8 % |     92 |
| YAML                |      6.296 |                +34,4 % |    279 |
| JSON                |      8.120 |                +73,4 % |    317 |
| XML                 |     10.815 |               +130,9 % |    352 |

**Hallazgo incómodo pero importante:** CORTEX no gana en caracteres frente a Markdown Tables en este corpus. Las tablas declaran las columnas una sola vez; CORTEX repite los atributos en cada entidad.

Sin embargo, CORTEX mantiene ventajas que Markdown pierde:

* una entidad completa por línea;
* atributos explícitos y localmente interpretables;
* mejor direccionamiento individual;
* menos dependencia de posición de columnas;
* mayor aptitud para composición, edición y procesamiento incremental.

Por tanto:

> **Markdown gana en compresión tabular homogénea. CORTEX ofrece mejor equilibrio entre densidad, semántica local y manipulabilidad.**

### 3. Inconsistencias encontradas

**La afirmación de que los seis formatos contienen “exactamente la misma información” es falsa.** 

Los principales problemas son:

1. **Texto plano se contradice sobre el deadline más cercano.** Declara que Horizon, con fecha 1 de agosto, es el más cercano, pero después muestra Atlas con fecha 30 de julio. 

2. **El texto plano agrega interpretaciones.** Por ejemplo, califica las relaciones Aegis–Phoenix como “dependencia circular”, algo que no se desprende inequívocamente de los registros y puede ser una duplicación semántica, no un ciclo. 

3. **I-004 no posee el mismo valor de duración:**

   * CORTEX: `duracion_h:0.0`
   * JSON/YAML/XML: nulo
   * Markdown: `—`
   * texto plano: omitido

   Cero horas y duración desconocida no son equivalentes.

4. **ComplianceAudit aparece como proveedor exclusivo de Phoenix, pero se lo responsabiliza por un incidente en Aegis.** La incoherencia existe incluso en las representaciones estructuradas. 

5. `post_incidente` contiene referencias a riesgos (`R01,R04` y `R05`), no solamente incidentes. El nombre del atributo contradice su contenido. 

6. El protocolo exige responder según §5 y emitir el formato de §6, pero **esas secciones no existen**.  El archivo termina inmediatamente después del XML. 

### 4. Problema metodológico central

La métrica “cuántos tokens inspeccionaste mentalmente” no es observable ni reproducible. Además, ordenar los seis formatos dentro del mismo prompt provoca:

* transferencia de memoria entre formatos;
* ventaja para los formatos leídos después;
* imposibilidad de aislar cuál representación produjo realmente la respuesta;
* ausencia de aleatorización;
* ausencia de múltiples ejecuciones y modelos;
* ausencia de verdad de referencia formal para las preguntas ambiguas.

El propio protocolo ordena leer primero las seis copias y después producir 48 respuestas, lo cual impide medir recuperación independiente por formato. 

## Veredicto

| Criterio                                | Ganador             |
| --------------------------------------- | ------------------- |
| Menor cantidad de caracteres            | **Markdown Tables** |
| Menor número de líneas                  | **CORTEX**          |
| Densidad con semántica local            | **CORTEX**          |
| Interoperabilidad existente             | **JSON**            |
| Lectura humana narrativa                | Plain Text          |
| Validación mediante ecosistemas maduros | JSON / XML          |
| Peor relación tamaño-información        | **XML**             |
| Mayor riesgo de deriva semántica        | **Plain Text**      |

**Conclusión para CODEC-CORTEX:** el corpus sí evidencia que CORTEX es mucho más compacto que JSON, YAML y XML, pero **no demuestra todavía una superioridad general frente a Markdown tabular ni una mayor precisión cognitiva**. Para demostrar eso hace falta ejecutar cada formato en prompts separados, con preguntas corregidas, orden aleatorio, respuestas canónicas y métricas observables.
