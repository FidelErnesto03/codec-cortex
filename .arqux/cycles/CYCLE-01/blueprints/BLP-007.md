# BLP-007: CE-2 — Implementacion Rust de canonicalizacion C14N-0.1
<!-- /BLP:TITLE -->

---

<!-- BLP:1 -->
## §1: Planteamiento del Problema

El Gate F3 (C14N-0.1) está bloqueado por CE-2: se requiere una segunda implementación independiente que produzca los mismos bytes canónicos que el oráculo Python para los 40 casos del corpus. El parser Rust de experiments/gate-f2-v2/rust/ ya parsea CORTEX 0.1 con 39/40 de precisión, pero no implementa el pipeline de canonicalización (orden canónico, normalización, serialización, hash).

Sin CE-2, el Gate F3 no puede avanzar, y la canonicalización sigue siendo un esfuerzo de una sola implementación — exactamente lo que la Constitution Art. XII (no autocertificación) busca evitar.

**Evidencia:**
- docs/review/GATE-F3-STATUS.md: CE-2 listado como BLOCKED_NOT_EXECUTED
- Python oracle: 40/40 golden bytes, 40/40 idempotence
- Rust parser existente: experiments/gate-f2-v2/rust/ con AST completo

**Impacto de no resolverlo:**
- Fase 3 queda en CORRECTED_DRAFT sin posibilidad de avanzar a ratified
- No hay forma de verificar que la canonicalización es determinista entre lenguajes
<!-- /BLP:1 -->

<!-- BLP:2 -->
## §2: Objetivo

Implementar un módulo de canonicalización Rust que extienda el parser existente de experiments/gate-f2-v2/rust/, produciendo bytes canónicos C14N-0.1 idénticos al oráculo Python para los 40 casos del corpus, desbloqueando CE-2 del Gate F3.
<!-- /BLP:2 -->

<!-- BLP:8 -->
## §8: Diseño Técnico

El módulo `canonicalize.rs` implementa el pipeline C14N-0.1 como una transformación del AST producido por el parser existente:

### Pipeline

```
AST (de main.rs) → canonicalize() → bytes canónicos UTF-8 → SHA256
```

### 1. Orden canónico
- **$0 metas:** `format` → `enum_*` (alfa) → `micro_*` (alfa) → `namespace_*` (alfa) → `extension_*` (alfa) → resto (alfa)
- **$0 símbolos:** alfabético por namespace+symbol
- **Pares attrs:** orden del contrato declarado en $0, seguido de extras alfabético
- **Secciones e Ideas:** preservar orden observado
- **Extensiones:** alfabético por namespace:id:version

### 2. Normalización
- **Microtokens:** expandir atom → valor completo del microtoken
- **Unicode NFC:** strings lógicos normalizados a NFC; bloque preserva verbatim
- **Números:** representación decimal exacta preservada
- **Cuerpo:** trim newline inicial/final; blank lines internos preservados
- **Bloque:** preservar verbatim; solo normalizar newline de transporte (CRLF→LF)

### 3. Serialización
- Sin whitespace extra, sin indentación canónica
- Sin comentarios
- Sin source preservation
- `\n` como terminador de línea

### 4. Hash
```rust
let canonical_hash = sha256(b"CORTEX-C14N-0.1\x00" + &canonical_bytes);
let input_hash = sha256(&raw_bytes);
```

<!-- BLP:15 -->
## §15: Riesgos

| Riesgo | Impacto | Mitigación |
|---|---|---|
| Rust parser no produce AST exactamente igual al Python | Alto — bytes canónicos difieren | Debuggear diferencias de parsing. El parser Rust 39/40 en F2, el caso V022 puede requerir ajuste. |
| Orden de attrs por contrato difiere del Python | Medio — discrepancia en serialización | El Python usa ordered_keys con META_ORDERS y SYMBOL_ORDER. Replicar exactamente en Rust. |
| Microtokens no se expanden igual | Medio — diferencia semántica | El Python expande por cada field. Rust debe recorrer igual. |
