# BLP-004: Correccion spec CORTEX 0.1 — comillas, atom, glossary-valid, handoff
<!-- /BLP:TITLE -->

---

<!-- BLP:1 -->
## §1: Planteamiento del Problema

El Mini Gate F2 (BLP-003) reveló 3 tipos de problemas en la especificación CORTEX 0.1 REAL que requieren corrección antes de avanzar a Fase 3 (canonicalización):

**1. Regla de comillas incorrecta (F2-F001):** La spec exige comillas dobles para todo campo tipo `text`, pero el diseño ideático de CORTEX solo necesita comillas en el foco. El resto de campos son atoms. El parser Go tropezó con esto (I046_unquoted_attrs_text).

**2. 6 ambigüedades reveladas por el parser Go:** I003, I004, I021, I036, I050, I055 fueron aceptados cuando deberían ser rechazados. La spec no es suficientemente precisa en esas reglas.

**3. Ausencia de nivel `glossary-valid`:** El pipeline de validación actual (syntax → structure → interpretation) no distingue errores de $0 de errores de Ideas. $0 debe validarse como paso intermedio explícito.

**Evidencia:**
- Mini Gate F2: Go parser 37/40 valid, 51/57 invalid — 9 discrepancias contra golden
- Hallazgo F2-F001 documentado durante construcción de mission.cortex
- El validador Python rechazó `90pct` como atom sin que la spec lo prohiba

**Impacto de no resolverlo:**
- Fase 3 (canonicalización) se construiría sobre una especificación con ambigüedades conocidas
- Implementadores futuros encontrarían las mismas fricciones que Go y Rust
- CORTEX como handoff entre agentes perdería confiabilidad
<!-- /BLP:1 -->

<!-- BLP:8 -->
## §8: Diseño Técnico

### 1. Regla de comillas (F2-F001)
El campo designado como `focus` en el sigilo (spec §11.5) es el único que DEBE ser string quoted. Todos los demás campos del contrato son atoms. El tipo `text` en el contrato informa al consumidor sobre la naturaleza semántica del campo, no impone un requisito sintáctico de comillas.

**Cambio en spec §14.2:** Agregar regla: "El foco declarado en el sigilo se escribe entre comillas dobles. Los demás campos se escriben como atoms."

### 2. Definición de atom (spec §17.2)
```text
atom = ALPHA / "_" / (ALPHA / "_" 0*31 (ALPHA / DIGIT / "_" / "." / "/" / ":" / "-"))
```
Un atom no contiene espacios, no comienza con dígito, máximo 32 caracteres.

### 3. Pipeline de validación (spec §24-25)

| Nivel | Requisito | Bloqueante |
|---|---|---|
| syntax-valid | Gramática correcta | — |
| **glossary-valid** | $0 completo y coherente | ⬅️ NUEVO |
| structure-valid | Ideas siguen contratos de $0 | Depende de glossary-valid |
| interpretation-complete | Extensiones requeridas soportadas | Depende de structure-valid |

### 4. Handoff (standard/GATE-F2.md)
Agregar criterio: "CORTEX debe ser auto-documentante para agentes. Un agente debe poder recibir un documento CORTEX como orden de misión, parsear su glosario e instrucciones, y ejecutar sin traducción externa."

## §15: Riesgos

| Riesgo | Impacto | Mitigacion |
|---|---|---|
| Cambiar regla de comillas rompe corpus existente | Alto — casos con quotes en campos no-foco fallan | Los casos del corpus se actualizan junto con la spec. Los quites en no-foco se eliminan. |
| Agregar nivel glossary-valid invalida parsers existentes | Medio — parsers existentes (Python, Go) necesitan actualización | El nivel es aditivo (no elimina syntax-valid). Los parsers pueden optar no implementarlo y quedarse en structure-valid parcial. |
| Go parser no se actualiza a tiempo | Bajo — el Go parser fue experimental | Los cambios se validan solo contra Python. Go queda como referente de experimento. |
