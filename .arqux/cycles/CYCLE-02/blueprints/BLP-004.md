# BLP-004: Asegurar que los 5 parsers (Python, Node.js, Go, Rust, Bash) procesen correctamente la sintaxis :CAPA en secciones CORTEX — $N:CAPA sin título, $N: Título:CAPA, y $0:CAPA — mediante corrección del parser Python + portabilidad a los demás lenguajes + suite de tests de conformancia cruzada.
<!-- /BLP:TITLE -->

---

<!-- BLP:1 -->
## §1: Planteamiento del Problema

BLP-003 implementó la sintaxis `:CAPA` en secciones CORTEX (`$N: Título:CAPA`) en ~438 archivos del proyecto, pero los parsers no la procesan correctamente:

1. **Parser Python** (referencia): El regex de secciones en `parser.py` no reconoce `$N:CAPA` sin título (ej. `$1:KNOW`). La línea cae al manejador de glosario y lanza `G004_GLOSSARY_DECLARATION_MUST_BE_ATTRS`. Solo 2/40 tests de conformancia HCORTEX pasan.
2. **Parser Node.js**: No modela el campo `capa` en su AST. `$0:KERNEL` se rechaza como "Content outside section".
3. **Parser Go**: No verificado; probablemente mismo problema.
4. **Parser Rust**: No modela `capa` en su struct `Section`.
5. **Parser Bash**: No extrae `:CAPA`.

**Evidencia:** Ejecución de conformancia Python: 40/40 C14N fallan, 38/40 HCORTEX fallan. Node.js: 40/40 C14N + 40/40 HCORTEX fallan.

**Impacto:** La sintaxis `:CAPA` existe en el corpus normativo y archivos del proyecto, pero ningún parser la interpreta correctamente — incoherencia crítica entre canon y realidad.
<!-- /BLP:1 -->

<!-- BLP:2 -->
## §2: Objetivo

Que los 5 parsers (Python, Node.js, Go, Rust, Bash) procesen la sintaxis `:CAPA` en secciones CORTEX — incluyendo `$0:CAPA`, `$N:CAPA` (sin título), y `$N: Título:CAPA` — con 100% de aprobación en la suite de conformancia cruzada.

<!-- BLP:4 -->
## §4: Principio Rector

AXM:capa_parsimony{La sintaxis :CAPA no altera la semántica del AST. $1:KNOW y $1 producen el mismo AST salvo por el campo capa. El parser no debe rechazar ni ignorar silenciosamente :CAPA.}
AXM:capa_default_data{Toda sección $N (N≥2) sin :CAPA explícita se asume DATA por defecto. Las secciones $0 y $1 son casos especiales: $0 es siempre KERNEL, $1 puede tener capa explícita o None. El parser no debe inventar capas — si el autor no declaró capa en $N (N≥2), el parser asigna DATA automáticamente.}
<!-- /BLP:4 -->

El ejecutor opera sobre el repositorio CODEC-CORTEX. Existen 5 implementaciones del parser CORTEX, cada una en su propio subdirectorio. La sintaxis `:CAPA` está definida en la gramática ABNF/EBNF, documentada en el glosario normativo, y presente en 438+ archivos del proyecto. El parser Python es la implementación de referencia. Los parsers Node.js, Go, Rust y Bash deben mantener paridad de funcionalidad con Python.

La suite de conformancia en `conformance/` contiene 40 casos C14N y 40 casos HCORTEX que ejercitan la sintaxis completa incluyendo `:CAPA`. Esta suite es el juez de aprobación.

<!-- BLP:7 -->
## §7: Reglas Obligatorias

1. Todo parser debe tener un campo `capa` en su modelo Section/SectionDef — `None` si no hay capa, o uno de los 6 valores canónicos
2. `$0` con `:CAPA` solo puede ser `$0:KERNEL` — cualquier otro valor en `$0` se rechaza con error
3. Los valores válidos de capa son exactamente: `KERNEL`, `CORE`, `KNOW`, `DATA`, `FLOW`, `CACHE`
4. `:CAPA` nunca altera la semántica del contenido — solo metadata
5. **Regla de default: toda sección $N con N≥2 que no tenga :CAPA explícita se asigna capa `DATA` automáticamente en el AST.** $0 no tiene default (es KERNEL o error). $1 no tiene default (None si no se declara). $2+ sin :CAPA → DATA.
6. La canonicalización debe preservar `:CAPA` en la salida (no colapsar ni eliminar)
7. HCORTEX roundtrip: CORTEX→HCORTEX→CORTEX debe preservar `:CAPA`
<!-- /BLP:7 -->

## §8: Diseño Técnico

### Modelo de datos

En cada parser, el `Section` debe incluir `capa: Optional[str]`:
```
struct Section:
  id: int
  title: Optional[str]
  capa: Optional[str]    # KERNEL|CORE|KNOW|DATA|FLOW|CACHE o None
  ideas: List[Idea]
```

### Regla de default

Tras identificar el `id` de la sección (antes o después de parsear `:CAPA` explícita):
- Si `id == 0` y no hay `:CAPA` → `capa = None` (el glossary starter no necesita capa)
- Si `id == 1` y no hay `:CAPA` → `capa = None` (sección $1 puede no tener capa)
- Si `id >= 2` y no hay `:CAPA` → `capa = "DATA"` (default automático)
- Si hay `:CAPA` explícita → usar ese valor

### Cambios por parser

**Python** (`parser.py` lines 249-262): El bloque de sección regular actualmente solo extrae capa si hay título. Agregar regex `$N:(CAPA)$` antes del bloque existente. Al crear la sección, aplicar regla de default para N≥2.

```python
mn = re.match(r"^\$([1-9][0-9]*):(KERNEL|CORE|KNOW|DATA|FLOW|CACHE)$", stripped)
if mn:
    sid = int(mn.group(1))
    capa = mn.group(2)
    current_section = Section(id=sid, title=None, ideas=[], capa=capa)
    doc.sections.append(current_section)
    in_glossary = False
    i += 1
    continue
```

Al final del bloque de sección (tanto con título como sin título), aplicar default:
```python
if current_section.capa is None and current_section.id >= 2:
    current_section.capa = "DATA"
```

**Node.js** (`src/parser.js`): Agregar `capa: null` al constructor Section. En `parseCortex`, agregar regex análogo. Aplicar default DATA para N≥2.

**Go** (`cortex/parser.go`): Agregar campo `Capa string` a `Section`. Default DATA para N≥2.

**Rust** (`src/parser.rs`): Agregar `capa: Option<String>` a `Section`. Default DATA para N≥2.

**Bash** (`lib/codec-cortex.sh`): Extraer `:CAPA` con regex. Default DATA para N≥2.
