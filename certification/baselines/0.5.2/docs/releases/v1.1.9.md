## Brechas cerradas

### Fix 1 — `recover` repara `$0` existente pero incompleto

**Problema:** v1.1.8 era segura porque retornaba `rc=1` cuando `$0` estaba presente pero omitía sigilos usados. Sin embargo, no reparaba el caso. Ejemplo:

```text
$0: GLOSSARY
# IDN | identity | attrs | B | Semantic | Identity

$1: CONTENT
IDN:package{name:"legacy"}
KNW:topic{topic:"x", content:"y", status:"current"}
```

**Solución:**

- Nueva función `_repair_incomplete_glossary()`.
- Auto-declara sigilos observados pero ausentes en `$0`.
- Usa metadata canónica cuando el sigilo es conocido (`KNW`, `RSK`, `AUD`, etc.).
- Usa metadata conservadora `attrs/M/Semantic` para sigilos no canónicos.
- Asegura tipos canónicos mínimos en `$0`.
- Emite diagnóstico `W012_INCOMPLETE_GLOSSARY_REPAIRED`.

**Verificación:**

```bash
cortex recover legacy.cortex --out fixed.cortex --embed-aud-rsk --format json
cortex verify fixed.cortex --strict
cortex render fixed.cortex --mode audit --profile full
```

Resultado esperado:

- `recover rc=0`
- `verify rc=0`
- `KNW:topic` visible en HCORTEX
- `W012_INCOMPLETE_GLOSSARY_REPAIRED` visible en JSON

### Fix 2 — `--embed-aud-rsk` traza reparación de `$0` incompleto

**Problema:** la reparación de un `$0` incompleto necesitaba traza embebida equivalente a la reconstrucción de glosario.

**Solución:**

Cuando `--embed-aud-rsk` está activo y hubo reparación de `$0` incompleto:

- Se inserta `AUD:recovery` con `event:"incomplete_glossary_repair"`.
- Se inserta `RSK:incomplete_glossary_repaired`.
- Se inserta `RSK:auto_declared_<sigil>` por cada sigilo auto-declarado.
- Si además hubo reconstrucción total o estado vivo movido desde `$0`, el evento combina los eventos reales.

Ejemplo de salida esperada:

```text
AUD:recovery{event:"incomplete_glossary_repair", ...}
RSK:incomplete_glossary_repaired{...}
RSK:auto_declared_knw{...}
```

---

### Fix 3 — sección libre sin techo artificial

**Problema:** v1.1.8 buscaba sección libre en rangos acotados. Era correcto para los casos auditados, pero mantenía un límite artificial.

**Solución:**

Nueva función `_first_free_recovery_section_id()`:

```text
$1, $2, $3, ... hasta encontrar una sección inexistente.
```

No contamina secciones existentes aunque ya existan `$1`, `$99`, `$100` o cualquier otra.

**Verificación adversarial:** archivo con `$1..$100` existentes y entradas operativas dentro de `$0`.

Resultado esperado:

```text
$101: RECOVERED CONTENT
```

---

### Fix 4 — demo v1.1.9 sin falso positivo de sentinel

**Problema:** el demo v1.1.8 podía construir un archivo con nombre inválido por usar `n/a` en el filename, y luego marcar PASS solo por `rc != 0`.

**Solución:**

`scripts/cortex_demo_v1_1_9.sh`:

- sanitiza nombres de archivo para sentinels (`n/a` → nombre seguro);
- exige que `verify` retorne non-zero;
- exige que la salida contenga `E034_CRITICAL_REQUIRED_FIELD_EMPTY`.

Esto elimina el falso positivo.

## Criterio de aceptación enterprise-candidate

v1.1.9 puede declararse como **enterprise-candidate controlado**, no necesariamente como `enterprise-stable` definitivo.

Razones:

1. Las brechas críticas detectadas en v1.1.8 fueron cerradas.
2. La recuperación ya cubre `$0` ausente, `$0` vacío, `$0` con memoria operativa y `$0` existente incompleto.
3. HCORTEX muestra contenido recuperado y trazas `AUD/RSK`.
4. Los tests pasaron en venv limpia.
5. El demo ya no tiene falso positivo por filename inválido.

Condición recomendada antes de `enterprise-stable`:

- ejecutar una ronda final de auditoría adversarial externa sobre corpus real/histórico;
- revisar claims de `STATUS.md`/`BENCHMARK.md` contra evidencia medida;
- validar compatibilidad con el `SKILL_canon.md` empaquetado.
