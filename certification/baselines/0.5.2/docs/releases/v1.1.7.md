## Brecha cerrada: null semántico

La brecha detectada en la octava verificación era que el literal `null` no se bloqueaba en todos los campos críticos. `OBJ:main{success:null}` y `CNST:self_contained{rule:null}` pasaban `verify --strict` porque el parser trataba `null` como el string `"null"` (no como Python `None`), y `_is_field_empty()` no detectaba el string `"null"` como vacío.

### P0-1 — String "null" tratado como vacío

`_is_field_empty()` ahora trata `"null"` (case-insensitive, con strip) como vacío. Esto cubre el caso donde el parser o el CLI producen el string `"null"` en lugar de Python `None`.

### P0-2 — OBJ.success:null, CNST.rule:null, RSK.mitigation:null, AUD.evidence:null bloqueados

Cubierto por P0-1: todos los campos requeridos de sigilos críticos ahora rechazan `null` (None o "null") vía E034_CRITICAL_REQUIRED_FIELD_EMPTY (non-bypassable).

### P0-3 — update --set field=null serializa None, no "null"

- `_parse_set_pairs()` ahora convierte `"null"`/`"None"` a Python `None`.
- `parse_attrs_body()` también convierte bare `null` a `None`.
- Resultado: `update --set rule=null` produce `rule:None` que es detectado por E034 y bloqueado (non-bypassable).

### P1-4 — recover crea sección dedicada RECOVERED CONTENT aunque $1 exista

recover ahora verifica si `$1` ya existe; si existe, usa `$99` para RECOVERED CONTENT, preservando la semántica de la sección `$1` original.

### P1-5 — recover añade RSK cuando se mueve FCS/OBJ/WRK/STP/NXT desde $0

recover emite `W011_RECOVERED_LIVE_STATE` cuando se mueven sigilos de estado vivo desde `$0`, advirtiendo que se activó estado recuperado.
